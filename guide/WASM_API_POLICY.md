# WASM API Design Policy

This document defines the policy for designing public TypeScript APIs that wrap WebAssembly (WASM) modules in Navara.

## Problem Statement

WASM classes generated by `wasm-bindgen` have manual memory management requirements. When Rust code frees memory, any JavaScript references to those WASM objects become invalid, leading to:

- Use-after-free errors
- Memory leaks if `.free()` is not called
- Unpredictable behavior when objects are garbage collected

## Policy: Do Not Expose WASM Classes in Public APIs

**Rule**: Public APIs in `@navara/three` and `@navara/three_api` MUST NOT expose WASM-generated classes directly to consumers.

### What This Means

1. **Do not re-export WASM classes** - Classes from `@navara/engine-api` should NOT be part of the public API.

2. **Do not return WASM classes from functions** - Public functions should return plain JavaScript objects or Three.js types, not WASM classes.

3. **Do not accept WASM classes as parameters** - Public functions should accept plain JavaScript objects or Three.js types.

### Current Implementation

The following types are now properly wrapped:

| WASM Class | TypeScript Type | Location |
|------------|-----------------|----------|
| `Window` | `Window` (type-only) | `@navara/core` |
| `Transform` | `Transform` (type-only) | `@navara/core` |
| `CameraFrustum` | `CameraFrustum` (type-only) | `@navara/core` |
| `Plane` | `Three.Plane` | `three` |
| `Ray` | `Three.Ray` | `three` |
| `LLE` | `LatLngHeight` | `@navara/core` |
| `EllipsoidGeodesic` | Wrapper class | `@navara/three_api` |
| `Vec2` | `Vec2` (type-only) | `@navara/core` |
| `Vec3` | `Vec3` (type-only) | `@navara/core` |

### Correct Pattern: Wrapper Functions

Instead of exposing WASM classes, create wrapper functions that:
1. Accept plain JavaScript objects or Three.js types as input
2. Internally create WASM objects as needed
3. Perform the operation
4. Convert results to plain JavaScript objects or Three.js types
5. Call `.free()` on any WASM objects created
6. Return the plain JavaScript/Three.js result

**Example - Geodetic to Vector3:**
```typescript
// Public function accepts plain object, returns Three.js type
export function geodeticToVector3(lle: LatLngHeight): Vector3 {
  // Create WASM object internally
  const wasmLLE = new LLE(lle.lat, lle.lng, lle.height);
  const wasmResult = geodeticToXyz(wasmLLE);

  // Convert to Three.js type
  const result = new Vector3(wasmResult.x, wasmResult.y, wasmResult.z);

  // Free WASM objects
  wasmResult.free();

  return result;
}
```

**Example - Ray/Plane Intersection:**
```typescript
// Returns Three.js Ray instead of WASM Ray
export function getPickRay(
  windowObject: WindowObject,  // Plain object type
  camera: PerspectiveCamera,   // Three.js type
  vec2: Vector2,               // Three.js type
): ThreeRay {                  // Returns Three.js type
  // Create WASM objects internally
  const window = new Window(windowObject.width, windowObject.height, windowObject.pixel_ratio);
  const transform = new Transform(...);
  const frustum = new CameraFrustum(...);

  const wasmRay = nvGetPickRay(window, transform, frustum, new Vec2(vec2.x, vec2.y));

  // Convert to Three.js Ray
  const origin = new Vector3(wasmRay.origin.x, wasmRay.origin.y, wasmRay.origin.z);
  const direction = new Vector3(wasmRay.direction.x, wasmRay.direction.y, wasmRay.direction.z);
  const threeRay = new ThreeRay(origin, direction);

  // Free WASM objects
  wasmRay.free();

  return threeRay;
}
```

**Example - Geodesic Calculations (Wrapper Class):**

When a WASM class precomputes values for optimization, create a wrapper class instead of separate functions:

```typescript
import { EllipsoidGeodesic as EllipsoidGeodesicImpl, LLE } from "@navara/engine-api";

export class EllipsoidGeodesic {
  private _raw: EllipsoidGeodesicImpl;

  constructor(start: LatLngHeight, end: LatLngHeight) {
    this._raw = new EllipsoidGeodesicImpl(
      new LLE(start.lat, start.lng, start.height),
      new LLE(end.lat, end.lng, end.height),
    );
  }

  get distance(): number {
    return this._raw.distance;
  }

  interpolatePoints(granularity?: number): LatLngHeight[] {
    const wasmPoints = this._raw.interpolateGeodeticPoints(granularity ?? null);
    const results = wasmPoints.map((lle) => ({
      lat: lle.lat, lng: lle.lng, height: lle.height,
    }));
    wasmPoints.forEach((lle) => lle.free());
    return results;
  }

  dispose(): void {
    this._raw.free();
  }
}
```

Usage:
```typescript
const geodesic = new EllipsoidGeodesic(start, end);
console.log(geodesic.distance);
const points = geodesic.interpolatePoints(1000);
geodesic.dispose(); // Free WASM memory when done
```

### Type Definitions

Use the `NormalizeWASMClass` utility to create TypeScript types from WASM class definitions:

```typescript
// web/navara_core/src/types/window.ts
import type { Window as EngineWindow } from "@navara/engine-api";
import type { NormalizeWASMClass } from "./utils";

export type Window = Required<NormalizeWASMClass<EngineWindow>>;
```

This utility:
- Removes the `free()` method
- Extracts properties from getter/setter methods
- Creates a plain object type

### Usage in Consumer Code

**Correct - Using plain objects:**
```typescript
import { type Window, getPickRay, convertWorldToScreen } from "@navara/three";

// Create plain object matching the Window type
const win: Window = { width: 800, height: 600, pixel_ratio: 2 };

// Use wrapper functions
const ray = getPickRay(win, camera, screenPos);  // Returns Three.js Ray
const worldPos = convertWorldToScreen(win, camera, point);  // Returns Vector2
```

**Incorrect - Trying to use WASM classes:**
```typescript
// This is no longer possible - WASM classes are not exported
import { Window } from "@navara/three";  // Error: Window is a type, not a class
const win = new Window(800, 600, 2);  // Error!
```

## Enforcement

- WASM classes are exported as `type` only, not as values
- Code review must verify no new WASM classes are added to public exports
- TypeScript will catch attempts to instantiate type-only exports

## Exceptions

Internal modules (not part of public API) may use WASM classes directly when:
- Performance is critical and object creation overhead is significant
- The WASM object lifecycle is fully managed within the module
- The module is clearly marked as internal (e.g., `@internal` JSDoc tag)
